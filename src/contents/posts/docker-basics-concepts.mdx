## 🐳 Docker란 무엇인가?

소프트웨어를 개발하고 운영하는 과정에서는 언제나 환경 문제라는 골칫거리가 생긴다.

개발자의 컴퓨터에서는 잘 돌아가던 코드가 서버에만 올리면 에러를 뿜는다거나, 운영 체제나 라이브러리 버전이 조금만 달라도 예상치 못한 문제가 발생한다.

Docker는 바로 이 문제를 해결하기 위해 등장한 기술이다.

Docker는 **컨테이너(Container)** 라는 가볍고 독립적인 실행 환경을 제공한다.

컨테이너는 애플리케이션 코드와 함께 해당 코드가 구동되기 위해 필요한 모든 라이브러리, 설정 파일, 종속성을 하나로 묶어 패키징하고, 이를 격리된 환경에서 실행할 수 있게 해준다.

> "내 컴퓨터에서 잘 됐던 그 상태 그대로"
>
>"어디서나, 누구나"
>
>"동일하게 실행할 수 있게"
> 

Docker는 2013년에 오픈소스로 공개되었고, 이후 전 세계 개발과 운영 분야에 혁신을 일으켰다.

지금은 단순히 개발 도구를 넘어서, 클라우드, DevOps, 마이크로서비스 아키텍처, 빅데이터 분야까지 널리 사용되고 있다.

## ❓ Docker가 필요한 이유

그럼 왜 굳이 Docker 같은 새로운 기술이 필요했을까?

소프트웨어는 단순히 코드만으로 동작하지 않는다.

운영체제(OS), 시스템 라이브러리, 패키지 버전, 환경변수 설정 등 수많은 요소들이 서로 얽혀 있다.

조금이라도 설정이 어긋나면 **환경 차이로 인한 문제**가 발생한다.

특히 서버가 늘어나고 서비스가 복잡해질수록 문제는 더 심각해진다.

다른 서버에는 설치되지 않은 라이브러리가 필요하거나, OS 패치 수준이 달라서 에러가 나는 상황이 빈번하게 발생한다.

이런 문제를 해결하기 위해서는:

- 애플리케이션과 필요한 모든 것을 **포함하고**
- **일관된 실행 환경**을 제공하고
- 어디서나 **쉽게 배포**할 수 있어야 한다.

Docker는 이를 완벽하게 만족시켜준다.

컨테이너라는 개념 덕분에 개발자는 **환경** 걱정 없이 코드에만 집중할 수 있고,

운영팀은 **서버 설정**에 신경 쓰지 않고 빠르게 배포하고 확장할 수 있게 된다.

결국 Docker는 **환경 문제를 제거하고 소프트웨어 전달 방식을 혁신하는 것**이 핵심 목표라고 할 수 있다.

## 🛠️ Docker의 주요 구성 요소

Docker는 하나의 기술이 아니라 여러 요소들이 맞물려서 동작한다.

Docker의 기본 구성 요소를 이해하면, 전체 시스템을 보다 쉽게 이해할 수 있다.

### 1️⃣ Docker Engine

Docker의 심장이다.

Docker Engine은 컨테이너를 생성하고 실행하고 관리하는 백그라운드 서비스다.

보통 서버나 개인 컴퓨터에 설치하고, CLI 명령어를 통해 제어하게 된다.

### 2️⃣ Docker Image

컨테이너를 만들어내는 **템플릿**이다.

Image 안에는 애플리케이션 실행에 필요한 모든 파일, 설정, 라이브러리 등이 포함되어 있다.

이미지는 읽기 전용(immutable)이며, 필요할 때 복제해서 여러 컨테이너를 만들 수 있다.

### 3️⃣ Docker Container

이미지를 실행시켜서 만들어지는 실제 **살아있는 인스턴스**다.

컨테이너는 격리된 공간 안에서 자신의 파일 시스템, 네트워크를 갖고 독립적으로 동작한다.

필요할 때 빠르게 생성하고, 삭제하고, 복제할 수 있다.

### 4️⃣ Dockerfile

이미지를 만들기 위한 레시피다.

어떤 OS를 사용할지, 어떤 패키지를 설치할지, 어떤 파일을 복사할지 등을 명령어 형태로 기술한다.

### 5️⃣ Docker Hub

Docker 이미지의 저장소다.

GitHub가 소스코드를 공유하는 곳이라면, Docker Hub는 이미지를 공유하는 곳이다.

공식 이미지나 커뮤니티 이미지들을 쉽게 검색하고 가져올 수 있다.

이렇게 각각의 구성 요소가 합쳐져서 Docker 생태계를 구성하고 있다.

## ⚙️ Docker의 내부 기술 원리

Docker는 단순히 파일을 묶어 실행하는 것처럼 보일 수도 있지만, 내부적으로 매우 복잡한 기술을 사용하고 있다.

가장 핵심적인 기술은 **Linux의 커널 기능**이다.

Docker는 커널의 기능을 이용해서 가상화 수준의 격리를 제공하지만, 실제로는 별도의 가상머신을 띄우지 않는다.

### 1️⃣ Namespace

Namespace는 시스템 자원(프로세스, 네트워크, 파일 시스템 등)을 **다른 컨테이너와 격리**해주는 기술이다.

덕분에 서로 다른 컨테이너가 같은 시스템 위에 있어도, 자신만의 독립적인 세상을 가진 것처럼 동작하게 된다.

### 2️⃣ Control Groups (cgroups)

cgroups는 컨테이너가 사용할 수 있는 **CPU, 메모리, 네트워크 대역폭** 같은 자원량을 제한하거나 관리할 수 있게 해준다.

이를 통해 하나의 컨테이너가 자원을 독식하거나 시스템 전체를 먹통 만드는 것을 방지할 수 있다.

### 3️⃣ Union File System (UnionFS)

Docker 이미지는 레이어(layer)라는 개념으로 관리된다.

기본 OS 레이어, 애플리케이션 레이어, 설정 파일 레이어 등 여러 개의 레이어가 쌓여서 하나의 이미지가 된다.

이 구조 덕분에 저장 공간을 절약할 수 있고, 중복 없이 빠르게 이미지를 복제할 수 있다.

이런 기술 덕분에 Docker는 **가볍고 빠르며** **격리된 환경**을 만들 수 있게 된다.

## 🔀 Docker와 가상머신(VM)의 차이점

Docker를 이해하려면, 전통적인 가상머신(VM)과 무엇이 다른지를 정확히 알아야 한다.

### 🔹 가상머신(VM)

- 하드웨어를 가상화한다.
- 각각의 VM은 **독립된 운영체제(OS)** 를 갖는다.
- 하이퍼바이저(VMware, Hyper-V 등)가 물리 서버 위에 VM을 관리한다.
- 부팅 속도는 느리고, 리소스 소모가 크다.

### 🔹 Docker

- 운영체제의 커널을 공유한다.
- 각각의 컨테이너는 **애플리케이션과 필요한 라이브러리만 포함**한다.
- 별도 운영체제가 없다.
- 컨테이너 시작 속도가 매우 빠르고, 리소스 사용이 가볍다.

| 항목 | Docker | 가상머신(VM) |
| --- | --- | --- |
| 가상화 대상 | OS 레벨(커널 공유) | 하드웨어 전체 |
| 부팅 시간 | 수 초 이내 | 수 분 이상 |
| 리소스 소비 | 적음 | 많음 |
| 배포 속도 | 매우 빠름 | 느림 |

Docker는 가상머신보다 훨씬 **가볍고, 빠르고, 효율적**인 대안으로 자리 잡게 된다.

## ✅ Docker의 장점

이제 Docker가 왜 이렇게 많은 사랑을 받는지 장점을 정리해보자.

### 1️⃣ 경량화

컨테이너는 별도의 운영체제를 갖지 않고 커널을 공유하기 때문에 훨씬 가볍다.

서버 한 대에 수십 개, 수백 개의 컨테이너를 띄울 수 있다.

### 2️⃣ 빠른 시작

컨테이너는 몇 초 만에 기동된다.

VM처럼 OS 부팅을 기다릴 필요가 없다.

### 3️⃣ 이식성

Docker 이미지만 있으면, 어떤 서버에서든 똑같이 실행할 수 있다.

운영체제, 환경 설정 차이로 인한 오류를 극적으로 줄일 수 있다.

### 4️⃣ 일관성

개발 환경, 테스트 환경, 운영 환경 모두 같은 이미지로 배포할 수 있다.

"내 컴퓨터에서는 잘 됐는데..."라는 말을 더 이상 듣지 않아도 된다.

### 5️⃣ 확장성

컨테이너는 필요할 때 빠르게 추가하거나 제거할 수 있다.

클라우드 환경에서도 오토스케일링이 매우 쉽다.

### 6️⃣ DevOps와 궁합

Docker는 CI/CD 파이프라인 구축과 매우 잘 맞는다.

코드 변경 → 빌드 → 테스트 → 배포까지 자동화할 때 필수 요소로 자리잡았다.

Docker는 단순한 가상화 기술을 넘어서,

**소프트웨어 개발과 배포, 운영 방식을 근본적으로 바꿔놓은 혁신적인 기술**이다.

이제는 "Docker를 쓰는 것"이 특별한 일이 아니라,
